=== 設定與管理



==== 自訂儀錶板

建立與您的工作負載和優先順序相符的 AI 驅動儀表板，以便輕鬆了解營運狀況。自訂儀表板讓您擺脫固定檢視的束縛，專注於最重要的資訊。

只需用簡單易懂的語言描述您想要追蹤的內容，AI 就會根據您的要求建立儀表板。例如，您可以追蹤災難復原狀態、跨區域的容量和成本、儲存保護合規性以及其他營運指標。

自訂儀表板為唯讀模式，因此無法變更您的儲存空間或工作負載設定。您 Workload Factory 帳戶中的任何使用者都可以查看和管理這些儀表板，方便團隊共享見解並保持協作。

link:learn-custom-dashboards.html["深入瞭解自訂儀表板"]



=== EDA 工作負載



==== 利用 IOPS 和吞吐量圖表增強延遲分析

延遲分析面板現在在「隨時間變化」標籤中顯示同步效能圖表：

* *延遲圖*：顯示延遲隨時間的變化情況，並帶有閾值線，當超過限制時會發出警示。
* *IOPS 圖表*：顯示每秒作業數，並附有閾值線和過去超出限制的指示器。
* *吞吐量圖*：顯示資料傳輸速率（DataReadBytes 或 DataWriteBytes）。


所有圖表均使用相同時間線，並根據事件是讀取還是寫入自動顯示正確的指標。

新增的篩選器可讓您在「全部」、「讀取」、「寫入」或「元資料」操作之間切換，以便進行更詳細的分析。

這些圖表可協助您了解延遲、IOPS 和處理量之間的關係，讓您更容易找出效能問題的原因並決定如何解決這些問題。

效能圖表目前僅在提供 AWS 憑證後可用。FSx for ONTAP 連結僅在基本分析和 AI 分析中的元件細分時才需要。

https://docs.netapp.com/us-en/workload-eda/task-analyze-latency-trends.html["學習如何使用多個圖表分析延遲趨勢"^].



==== 設定延遲通知頻率

現在您可以選擇接收延遲違規電子郵件通知的頻率：

* *每天提醒我（建議）*：當任何 FSx for ONTAP 檔案系統出現新問題時，您將收到通知。


如果一天內同一檔案系統上發生多個延遲問題，您只會收到一封電子郵件。

Workload Factory 在向檔案系統發出延遲警報後，會等待 24 小時，再向該檔案系統發出另一個警報。

* *Notify on any latency breach (every 20 minutes)*：偵測到違規情況時，最多每 20 分鐘收到一次通知。


預設啟用每日通知，以減少電子郵件數量，同時仍能提醒您注意延遲問題。您可以在設定延遲監控或編輯閾值設定時變更通知偏好設定。

https://docs.netapp.com/us-en/workload-eda/task-configure-latency-monitoring.html["了解如何設定延遲監控"^].

==== 在 NetApp Workload Factory 中追蹤複寫延遲

借助 NetApp Workload Factory，您可以追蹤配置了遠端複製的 FSx for ONTAP 檔案系統的複製延遲。複製延遲是指資料寫入來源檔案系統到複製到目標檔案系統之間的時間延遲。追蹤複製延遲有助於您監控複製設定的運作狀況，並識別可能影響資料一致性和復原點目標（RPO）的潛在問題。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/track-replication-delays.html["在 NetApp Workload Factory 中追蹤複寫延遲"]



==== 在 NetApp Workload Factory 中的主機識別與工作負載定義

Workload Factory 可辨識使用儲存的主機以及與主機活動相關的工作負載類型。工作負載定義由 AI 根據 AWS 標籤和中繼資料產生。您可以在 Workload Factory 的檔案系統、磁碟區和區塊裝置資源畫面中查看主機詳細資訊和工作負載定義。

我們在檔案系統的「概覽」標籤中直接顯示所有檔案系統和儲存 VM 端點。在 Workload Factory 主控台中直接查看端點，無需使用 CLI 或 CloudShell 即可輕鬆監控網路存取、端點和容錯移轉狀態。若要顯示 LIF 詳細資訊（包括角色、端點、節點、IP 位址和狀態），您需要將連結與檔案系統建立關聯。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/create-link.html["了解如何在 NetApp Workload Factory 中將連結與檔案系統建立關聯。"]。



==== 成本增強功能

新增的選用頁面層級篩選器可最佳化跨所有 AWS 帳戶的資源成本分析結果。您可以選擇多個區域，並使用檔案系統指派標籤鍵和鍵值來篩選成本分析結果，從而取得 FSx for ONTAP 檔案系統和磁碟區的特定資源成本詳情。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/track-costs.html["在 NetApp Workload Factory 中使用篩選器進行成本分析"]

在資料複製設定過程中，您現在可以為 FSx for ONTAP 檔案系統排程複製初始化。您可以選擇立即初始化複製關係，也可以選擇在指定時間初始化。複製初始化是透過將資料從來源檔案系統複製到目標檔案系統來建立複製基線的過程。安排複製初始化可以自動執行此過程，並確保您的複製設定無需人工幹預即可保持最新狀態。

link:https://docs.netapp.com/us-en/workload-fsx-ontap/create-replication.html["在 NetApp Workload Factory 複製資料"]

==== 檢視和分析延遲事件

Workload Factory 現在支援檢視和分析 FSx for ONTAP 磁碟區的延遲事件。您無需連結連線即可檢視延遲事件，從而更輕鬆地監控效能問題。若要分析延遲事件並取得 AI 驅動的洞察和建議，您必須使用連結連線到檔案系統。分析結果提供目前延遲值、臨界值、QoS 回應分析和補救指南。

* *建議*

目前模式及其參數顯示在檔案系統詳細資料頁面的 SSD 使用情況圖表頂部。您可以根據實際操作需要隨時切換模式。

==== 批次編輯 SSD 容量增加管理參數

現在，您可以從概覽儀表板跨多個檔案系統調整 SSD 容量增加管理參數。在「叢集」表中選擇一個或多個叢集，然後批量編輯臨界值（10-90%）、回溯（1-200 小時）和預估（1-200 小時）的值。

編輯對話方塊會根據您的選擇自動填入：如果所有選定的檔案系統都具有相同的參數值，則會顯示這些值；否則，將使用預設值。這有助於您在整個環境中應用一致的容量設定。

link:https://docs.netapp.com/us-en/workload-eda/task-use-overview-dashboard.html#bulk-edit-ssd-management-parameters["了解如何批次編輯參數"^]。



==== 容量決策點視覺化

SSD 使用情況圖表現在會顯示決策點，指示何時產生容量建議或採取自動化動作。

* *建議決策點*：將滑鼠懸停可檢視時間戳記、建議增加的大小和目前容量。
* *自動化決策點*：滑鼠懸停可檢視時間戳記、成功或失敗狀態、增加大小以及新的 Aggregate 大小。失敗的嘗試包括失敗原因。


決策點每 30 分鐘出現一次，並根據所選時間範圍單獨顯示或合併顯示。

link:https://docs.netapp.com/us-en/workload-eda/task-use-overview-dashboard.html#analyze-capacity-trends-and-decision-points["了解容量決策點"^]。



==== 叢集表格中的連結已設定狀態

==== 延遲事件的電子郵件和 SNS 通知

現在，當 FSx for ONTAP 磁碟區上偵測到延遲事件時，您可以接收自動通知。在 Workload Factory 設定中設定電子郵件或 Amazon SNS 通知，以便在磁碟區的延遲超過您設定的閾值時接收即時警報。

通知會以檔案系統為基礎傳送，包括所有受影響磁碟區的詳細資訊、嚴重性層級、臨界值比較，以及可直接連結至 Latency 頁面進行調查的連結。

https://docs.netapp.com/us-en/workload-eda/task-monitor-latency.html#configure-latency-notifications["了解如何設定延遲通知"^]。



==== 使用互動式圖表將延遲趨勢視覺化

延遲分析面板現在包含分頁式介面，可進行全面的延遲分析：

* *概述*：檢視自動化基本分析結果，顯示哪個元件造成延遲
* *隨時間變化*：透過互動式圖表視覺化多個時間範圍（1 小時、3 小時、12 小時、24 小時或 72 小時）內的磁碟區延遲行為


*Over time* 標籤顯示原始 CloudWatch 延遲資料（讀取或寫入，取決於觸發的警示），以及警告和嚴重臨界值線、違規指標和每次違規的詳細指標。

這種全面的視覺化功能可協助您識別模式、瞭解延遲問題是反覆出現還是孤立發生，並針對補救措施做出更明智的決策。

https://docs.netapp.com/us-en/workload-eda/task-analyze-latency-trends.html["瞭解如何分析延遲趨勢"^]。
